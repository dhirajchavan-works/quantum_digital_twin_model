## The problem this layer solves: drift

A digital twin that only runs from a physics model and never receives real data from the actual ship will slowly become wrong over time. The physics equations are accurate on average, but every real ship has manufacturing variations, coating batches that behave slightly differently from spec, and unusual environmental exposures the model did not anticipate.

This drift gets worse year after year. By year three or four, the twin's predictions could be significantly off from reality — and the fleet engineer would not know to distrust them. The integration layer prevents this by providing **real-world corrections** that keep the twin calibrated to the actual ship.

## Three Input Channels — how reality talks to the twin

### Input Channel 1 — Onboard Hull Sensors

Ships can be fitted with electronic sensors mounted directly on the hull. These sensors measure the actual physical state of the hull in real time and send data back to the twin automatically.

| Sensor Type | What It Measures | How Often | Feeds Into |
|-------------|-----------------|-----------|-----------|
| Corrosion probe (LPR) | Active corrosion rate at probe | Hourly | `corrosion_rate` |
| Ultrasonic thickness gauge | Actual coating + metal thickness | Daily | `coating_thickness` |
| Hull-mounted camera | Visual fouling (computer vision) | On demand | `barnacle_density` |
| Water quality sensor | Temperature, O₂, salinity at hull | Continuous | `local_*` variables |
| Speed + fuel monitor | Ship speed, fuel burn rate | Continuous | `flow_velocity`, `fuel_penalty` |

**Data path:**
```
Hull sensor → ship-side edge gateway → satellite / port WiFi
    → shore ingestion server → state corrector → zone state update in twin
```

### Input Channel 2 — Physical Inspection Reports

Every time the ship goes into drydock or gets an in-water inspection, a human inspector physically measures the hull. These measurements are **ground truth**. When an inspection report is uploaded, the twin runs a state correction pass that overrides its stored values with the actual measured ones.

```python
def apply_inspection_correction(inspection_data, inspection_date):
    for zone_id, measured in inspection_data.items():
        zone = ship.get_zone(zone_id)

        # Override simulation state with measured ground truth
        zone.corrosion_depth  = measured['corrosion_mm']
        zone.coating_thickness = measured['coating_um']
        zone.barnacle_density  = measured['fouling_count_per_m2']
        zone.last_validated    = inspection_date

        # Check if the simulation was drifting from reality
        drift = abs(zone.corrosion_depth - previous_sim_value)
        if drift > DRIFT_TOLERANCE:
            flag_model_for_recalibration(zone_id)

# Accepted formats: drydock CSV, ROV survey PDF, UT scan XML
```

### Input Channel 3 — Satellite and Marine Environmental Data

For any ship sailing anywhere in the world, the twin needs ocean data for that position. This comes from a daily automated ETL pipeline.

```
Daily ETL Pipeline (runs automatically at 02:00 UTC every day):

  EXTRACT:
    CMEMS API (Copernicus Marine Service)
      → sea temperature, salinity, dissolved O₂, ocean currents (NetCDF)
    NOAA / ECMWF
      → 48-hour ocean and weather forecasts
    AIS (Automatic Identification System) feed
      → ship GPS positions, speeds, port arrival and departure events
    Port authority databases
      → idle time at each port per vessel ID

  TRANSFORM:
    Regrid NetCDF files from 0.25-degree global grid
      → zone-mapped values using bilinear interpolation
    Join AIS data with ship zone positions
    Calculate hours_in_harbour per timestep from port events

  LOAD:
    env_database (PostgreSQL + PostGIS)
      → ready for simulation Step 1 to query in real time
    Data retention: 5 years for historical analysis
```
