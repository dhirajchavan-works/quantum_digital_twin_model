## PHASE 01 — Digital Twin Definition
*`digital_twin_definition.md`*

### What is a digital twin?

A digital twin is a software copy of something real that keeps updating itself over time. Not a static model you build once and forget. Not a diagram on paper. An actual running program that continuously simulates what is happening to the real object.

For a ship, the digital twin is a running simulation of the ship's hull — specifically, how the hull degrades over time. The real ship is out at sea getting rusty, collecting barnacles, and losing its protective paint. The twin is doing the exact same thing in software, on a timeline we control. We can run it at real speed, speed it up, pause it, or look years into the future.

Why do we need this? Because physically inspecting a ship is expensive and slow. You have to bring the ship to a drydock, which costs money and takes it out of service. With a working digital twin, the engineer just opens a laptop and asks the simulation questions instead.

| Real Ship | | Digital Twin |
|-----------|---|-------------|
| Rusts in seawater | ↔ mirrors | Simulates rusting |
| Barnacles grow | ↔ mirrors | Simulates growth |
| Paint wears off | ↔ mirrors | Simulates wear |

**What the twin tells you (without touching the real ship):**
- How thick is the coating on Zone 4 right now?
- When will barnacles push fuel cost up by 5%?
- Which zones are dangerously corroded in the next 3 months?

### What exactly does the twin mirror?

After going through the physics carefully, four main processes drive all the damage and cost on a ship hull.

#### 1. Corrosion — the metal is slowly eating itself

Seawater is an electrolyte — it conducts electricity. When steel is submerged in it, an electrochemical reaction happens: iron atoms slowly convert to iron oxide (rust) and dissolve away. Speed depends on dissolved oxygen (more O₂ = faster corrosion), salinity, and temperature. The most important protection is the paint coating — when intact, corrosion is almost completely stopped. When the paint fails, corrosion accelerates rapidly.

#### 2. Biofouling — organisms colonise the hull

As soon as a ship is in water, marine organisms start attaching. It starts with a thin biofilm, then algae and barnacles. Barnacles create a rough, bumpy surface that dramatically increases resistance. More resistance means more fuel burned to maintain speed. A heavily fouled ship can use **10–15% more fuel** than a clean one. Fouling grows fastest in warm tropical water and when the ship is idle in harbour.

#### 3. Coating degradation — the paint protection wears away

Three things wear down the anti-corrosion and anti-fouling paint:
- High-speed water flow causes **mechanical erosion** (especially at the bow)
- UV radiation **breaks down the polymer chemistry** above the waterline
- Saltwater **chemically degrades** the paint from below

Once the paint gets thin enough, both corrosion and fouling accelerate. Coating is the master variable — everything else in the simulation depends on its status.

#### 4. Performance loss — drag costs money

As the hull surface gets rougher from corrosion pitting and barnacle growth, hydrodynamic drag increases. More drag = more fuel burned to maintain speed. The simulation calculates this drag increase, converts it to a fuel penalty percentage, and then to an annual dollar figure.

### The state variables — what numbers the twin tracks

| Variable | Unit | Plain English Meaning | Used In |
|----------|------|----------------------|---------|
| `corrosion_depth` | mm | How many mm of steel have been eaten away | Risk score, maintenance schedule |
| `coating_thickness` | µm | How much paint is still on the surface | Controls corrosion + fouling rate |
| `barnacle_density` | count/m² | How many barnacles per square metre | Drag calculation, fuel penalty |
| `flow_velocity` | m/s | How fast water flows over that hull section | Coating erosion rate |
| `water_temperature` | °C | Sea temperature at the hull | Corrosion k, fouling growth rate |
| `oxygen_concentration` | mg/L | Dissolved O₂ in the seawater | Corrosion rate equation |
| `salinity` | PSU | How salty the water is | Electrochemical potential factor |
| `roughness_index` | Ra µm | Surface texture — corrosion + fouling combined | Drag coefficient |
| `risk_score` | 0.0–1.0 | Aggregated danger level for this zone | Alerts, dashboard colour map |

### What is deliberately left out of scope

To keep this buildable and focused, the twin does **NOT** model: internal structural fatigue, propulsion system wear, cargo load effects, or GPS routing. These could be added as separate modules later. The current twin is focused entirely on hull surface degradation. Keeping the scope tight is what makes it deployable.

## PHASE 02 — State Representation Model
*`state_model.md`*

### The first design decision: should the hull be one thing or many?

The answer becomes obvious when you think about what actually happens on a real ship. The bow crashes into waves at high speed — maximum mechanical stress and coating erosion. The keel sits in slow-moving deep water — low flow but lots of biofouling. The waterline gets UV from the sun and the heaviest barnacle growth. Averaging all of this together gives meaningless numbers.

So the design is a **zone-based grid**. The hull is divided into 50–200 rectangular patches called zones. Each zone is an independent unit with its own state, updated separately every timestep.

```
BOW ─────────────────────────────────── STERN

WATERLINE ROW
  Z01   Z02   Z03   Z04   Z05   Z06   Z07   Z08

MID-DEPTH ROW
  Z09   Z10   Z11   Z12   Z13   Z14   Z15   Z16

KEEL ROW
  Z17   Z18   Z19   Z20   Z21   Z22   Z23   Z24
```
*Each zone stores its own state variables: corrosion_depth · coating_thickness · barnacle_density · flow_velocity · roughness_index · risk_score*

### The zone data structure

Every zone in the grid is an instance of this data structure. Every single field either comes from a real measurement, gets computed from other fields, or feeds into another calculation.

```python
Zone {
    # Identity fields
    zone_id        : string   -- e.g. 'Z_045'
    position       : (x, y, z) -- metres from ship reference point
    area           : float    -- m², used in drag and fuel calculations
    zone_type      : string   -- bow / midship / stern / keel / waterline

    # Corrosion (how deep has the metal been eaten)
    corrosion_depth : float   -- mm of total metal lost so far
    corrosion_rate  : float   -- mm/year, recomputed each timestep

    # Coating (is the paint still protecting the hull)
    coating_thickness : float   -- micrometres (µm) remaining
    coating_intact    : boolean -- True if thickness > failure threshold (default 15 µm)
    coating_type      : string  -- e.g. 'epoxy_antifouling_v2'

    # Biofouling (what has attached to the hull surface)
    barnacle_density : float  -- count per m²
    slime_thickness  : float  -- micrometres of biofilm
    fouling_stage    : string -- clean / slime / barnacle / heavy

    # Environment (refreshed every timestep from the environment database)
    local_flow_velocity : float -- m/s
    local_temperature   : float -- degrees Celsius
    local_oxygen        : float -- mg/L dissolved oxygen
    local_salinity      : float -- PSU practical salinity units

    # Computed outputs (calculated at end of each timestep)
    roughness_index   : float -- Ra µm, from corrosion + fouling
    drag_coefficient  : float -- dimensionless, drives fuel penalty
    risk_score        : float -- 0.0 (healthy) to 1.0 (critical)
    last_validated    : date  -- date of last real inspection correction
}
```

### The four update rules — how zone state changes over time

These rules are not independent — they form a chain. Rule 1 is the most important because it controls everything else.

```
RULE 1  Coating = Master Switch
        If coating is intact  →  corrosion slows 95%, fouling slows 90%
        If coating fails      →  both run at full speed
            ↓
RULE 2  Corrosion → Roughness
        Every mm of metal lost makes surface rougher
        roughness_index = corrosion_depth × roughness_factor
            ↓
RULE 3  Fouling → Drag
        More barnacles = more drag = more fuel burned
        drag = base_drag × (1 + barnacle_density × drag_per_barnacle)
            ↓
RULE 4  Flow → Coating Wear
        Faster water flow erodes paint faster
        coating_loss = erosion_k × velocity² × time_step
            ↺ (Feedback loop back to Rule 1)
```

#### Rule 1 — Coating is the master switch

```python
IF zone.coating_intact == True:
    corrosion_rate  = base_corrosion_rate * 0.05   # coating blocks 95% of corrosion
    biofouling_rate = base_fouling_rate   * 0.10   # antifouling paint blocks 90% of growth
ELSE:
    corrosion_rate  = k * oxygen * salinity_factor * temperature_factor
    biofouling_rate = base_rate * temperature_factor * harbour_idle_factor

# k is the material rate constant from the quantum parameter table (material_table.json)
```

#### Rule 2 — Corrosion makes the surface rougher

```python
zone.roughness_index += zone.corrosion_depth * roughness_factor
# roughness_factor is material-specific, comes from the quantum material table
```

#### Rule 3 — Biofouling increases drag (and fuel cost)

```python
zone.drag_coefficient = base_drag * (1 + zone.barnacle_density * drag_per_barnacle)
# Fouling stage depends on barnacle count:
# 0 → clean | 0–10/m² → slime | 10–100/m² → barnacle | 100+/m² → heavy
```

#### Rule 4 — Fast water flow erodes the coating

```python
mechanical_loss = erosion_constant * flow_velocity**2 * time_step
chemical_loss   = coating_chemical_decay_rate * time_step
uv_loss         = uv_decay_rate * uv_index * time_step  # above waterline only

zone.coating_thickness -= (mechanical_loss + chemical_loss + uv_loss)
zone.coating_intact     = (zone.coating_thickness > 15)  # 15 µm = failure threshold
```

### How often does the simulation update?

| Mode | Timestep | Steps/Year | Best For | Trade-off |
|------|----------|-----------|---------|-----------|
| Fast preview | Monthly | 12 | 5–10 year strategic planning | Misses short weather events |
| Standard (default) | Weekly | 52 | Normal operational forecasting | Good balance of speed + accuracy |
| High fidelity | Daily | 365 | Deep investigation of one zone | Slow but very accurate |

## PHASE 03 — Simulation Engine Design
*`simulation_engine.md`*

### The engine is a loop — and the order of steps matters

The simulation engine is a time-loop: every timestep, it runs through five steps in a fixed order, then advances the clock by one timestep (one week, by default), then repeats. **The order cannot be changed** — each step uses outputs from the previous step.

```
STEP 1  Input Update — Environment
        Pull sea temperature, salinity, dissolved O₂ from database. Assign to each zone.
        Inputs: CMEMS satellite data, AIS ship speed
            ↓
STEP 2  CFD Update — Water Flow
        Calculate how fast water is moving over each part of the hull.
        Uses: pre-computed flow table indexed by ship speed + zone type
            ↓
STEP 3  Biofouling Growth
        Grow barnacles and biofilm on each zone based on temp, idle time, coating state.
        Warm water + harbour = fastest growth; intact coating = 90% suppression
            ↓
STEP 4  Corrosion Update
        Calculate metal loss per zone. Rate uses k value from quantum table.
        Coating protection = coating_thickness / max_thickness (0.0 to 1.0)
            ↓
STEP 5  Coating Degradation
        Wear down paint from flow erosion + UV + chemical decay. Flip intact flag.
        ↺ Step 5 output gates Steps 3 and 4 in the NEXT cycle
```

### Step 1 — Input Update

```python
for zone in ship.all_zones:
    zone.local_temperature = env_database.get_temperature(zone.position, current_date)
    zone.local_salinity    = env_database.get_salinity(zone.position, current_date)
    zone.local_oxygen      = env_database.get_oxygen(zone.position, current_date)
# Data source: CMEMS satellite data, preprocessed daily by ETL pipeline
# ship speed also updated here from AIS feed
```

### Step 2 — CFD Update

```python
# For live simulation: fast table lookup
zone.local_flow_velocity = flow_table[current_ship_speed][zone.zone_type][zone.position]

# For offline deep analysis: actual CFD formula
zone.local_flow_velocity = ship_speed * zone_geometry_factor * turbulence_coefficient
# flow_table pre-computed by OpenFOAM for speeds 0–25 knots, all zone types
```

### Step 3 — Biofouling Growth

```python
# Temperature factor: warmer water = faster growth (Arrhenius relationship)
temp_factor = exp(0.08 * (zone.local_temperature - 15.0))

# Harbour factor: ship idle in port gets more fouling than a ship underway
idle_factor = 1.0 + 0.3 * hours_in_harbour / 24.0

# Coating factor: intact antifouling paint suppresses 90% of fouling
coating_factor = 0.10 if zone.coating_intact else 1.0

# Final growth rate and density update
growth_rate          = BASE_FOULING_RATE * temp_factor * idle_factor * coating_factor
zone.barnacle_density += growth_rate * delta_t_days

# Classify fouling stage for the decision engine
if   zone.barnacle_density == 0:  zone.fouling_stage = 'clean'
elif zone.barnacle_density < 10:  zone.fouling_stage = 'slime'
elif zone.barnacle_density < 100: zone.fouling_stage = 'barnacle'
else:                             zone.fouling_stage = 'heavy'
```

### Step 4 — Corrosion Update

```python
# Get the rate constant k (from quantum-precomputed material_table.json)
k = material_table[zone.alloy_type][zone.local_temperature][zone.local_salinity]

# Coating protection: 1.0 = fully coated, 0.0 = bare metal
coating_protection = zone.coating_thickness / MAX_COATING_THICKNESS

# Corrosion rate equation (electrochemical model)
zone.corrosion_rate = k * zone.local_oxygen * salinity_factor * temp_factor \
                        * (1.0 - coating_protection)

# Update cumulative metal loss
zone.corrosion_depth += zone.corrosion_rate * delta_t_years
```

### Step 5 — Coating Degradation

```python
# Three coating wear mechanisms running simultaneously
mechanical_loss = EROSION_CONSTANT * zone.local_flow_velocity**2 * delta_t
chemical_loss   = coating_props['chemical_decay_rate'] * delta_t
uv_loss         = coating_props['uv_decay_rate'] * uv_index * delta_t
# uv_loss only applies to zones above the waterline

zone.coating_thickness = max(0.0, zone.coating_thickness
                             - mechanical_loss - chemical_loss - uv_loss)

# Update the gate flag — this is what Rule 1 checks in Steps 3 and 4
FAILURE_THRESHOLD  = coating_props['failure_threshold']  # 15 µm default
zone.coating_intact = (zone.coating_thickness > FAILURE_THRESHOLD)

# Compute derived outputs for the snapshot
zone.roughness_index  = zone.corrosion_depth  * roughness_factor \
                      + zone.barnacle_density * barnacle_roughness_factor
zone.drag_coefficient = BASE_DRAG * (1 + zone.barnacle_density * DRAG_PER_BARNACLE)
zone.risk_score       = compute_risk_score(zone)
```

### Layer dependencies — why you cannot reorder the steps

```
Step 1: Environment  →  provides local_temperature, local_oxygen, local_salinity
            ↓
Step 2: CFD Flow     →  provides local_flow_velocity (uses ship_speed from Step 1)
            ↓
Step 3: Biofouling   →  uses temperature, idle_time, coating_intact (from PREVIOUS cycle)
            ↓
Step 4: Corrosion    →  uses k table, oxygen, salinity, coating_protection
            ↓
Step 5: Coating      →  uses flow_velocity (Step 2), UV data
            ↓
        Updates coating_intact flag
            ↓
        FEEDS BACK into Steps 3 and 4 next cycle
            ↓
        Compute risk_score, drag, roughness → save snapshot
```

## PHASE 04 — Hybrid Compute Layer (Quantum + Classical)
*`compute_layer.md`*

### Why I split quantum and classical — the honest answer

Quantum computers today are excellent at a specific type of problem: computing the **exact electronic structure of materials**. They are NOT good at running a real-time iterative simulation with thousands of zone updates per timestep.

So the correct architecture is: **quantum runs once (or occasionally)** to produce highly accurate parameter tables, and **classical runs the live simulation every timestep** using those tables. The two layers are completely decoupled. You can upgrade one without breaking the other.

```
┌─────────────────────────────┐     ┌──────────────────────────────┐
│   OFFLINE — QUANTUM         │     │   ONLINE — CLASSICAL         │
│   (Runs once / when needed) │     │   (Runs every timestep)      │
│                             │     │                              │
│  VQE — Variational Quantum  │     │  Python: NumPy + SciPy       │
│  Eigensolver                │     │  All 5 simulation steps      │
│  → material_table.json      │     │  Zone state updates          │
│                             │     │  Reads quantum tables        │
│  Molecular Simulation       │     │                              │
│  → coating_properties.json  │     │  PostgreSQL + HDF5           │
│                             │     │  Zone state snapshots        │
└──────────────┬──────────────┘     │  Historical trend data       │
               │  DATA BRIDGE       └──────────────────────────────┘
               │  material_table.json
               └─ coating_properties.json
```

*At simulation startup: engine loads both JSON files into memory.*
`k = material_table[alloy][temperature][salinity]`

### OFFLINE QUANTUM — Part 1: Material Properties (VQE)

The corrosion rate constant **k** is the most physics-sensitive number in the simulation. VQE solves this exactly by directly simulating the quantum mechanics of the electrons in the material.

```json
// material_table.json — output of quantum VQE computation
{
  "steel_grade_DH36": {
    "corrosion_k": {
      "temp_10_salinity_30": 0.0318,
      "temp_15_salinity_33": 0.0374,
      "temp_20_salinity_35": 0.0421,
      "temp_25_salinity_35": 0.0589,
      "temp_30_salinity_38": 0.0712
    },
    "roughness_factor":  0.034,
    "failure_depth_mm":  3.0
  },
  "steel_grade_A36": { "..." : "..." }
}
```

### OFFLINE QUANTUM — Part 2: Coating Chemistry Parameters

```json
// coating_properties.json — output of quantum molecular simulation
{
  "epoxy_antifouling_v2": {
    "uv_decay_rate":       0.0032,
    "chemical_decay_rate": 0.0011,
    "failure_threshold":   15,
    "max_coating_thickness": 200,
    "adhesion_decay_curve": [1.0, 0.92, 0.81, 0.67, 0.48, 0.29]
  },
  "vinyl_tar_epoxy_v1": { "..." : "..." }
}
```

### ONLINE CLASSICAL — every timestep

When the simulation is running, there is no quantum hardware involved at all. The classical engine reads the JSON files at startup and uses them as lookup tables:

- **Python (NumPy, SciPy)** — all numerical zone update calculations
- **Pandas** — managing zone objects, writing snapshots to disk
- **OpenFOAM** — optional, used offline to pre-compute the flow table for Step 2
- **PostgreSQL + PostGIS** — environment database, spatial queries by zone position
- **HDF5** — efficient storage of zone state snapshots (time-series format)

### When does quantum need to run again?

| Event That Triggers a Re-Run | What Quantum Does | How Often |
|------------------------------|-------------------|-----------|
| New steel alloy used on the ship | Re-run VQE for that alloy | On demand |
| New coating product deployed | Re-run molecular sim for new coating | On demand |
| Annual accuracy audit fails | Full recalibration run | Once per year |
| Inspection shows model is drifting | Selective k-value correction run | As needed |
| Quantum hardware upgraded | Full table refresh for better accuracy | Once per gen |

**Hardware setup:** Currently using Qiskit (IBM) or PennyLane, running on IBM Quantum systems or classical simulators. The interface is designed so that when proper fault-tolerant quantum computers exist, we can point the system at real QPU hardware without changing any other code.

## PHASE 05 — Decision Engine
*`decision_engine.md`*

### Why this phase matters more than the simulation itself

All the work in Phases 1–4 produces numbers. But a fleet manager doesn't want to look at tables of numbers — they want to know: **what do I need to do, and when?**

The decision engine converts simulation outputs into concrete, actionable instructions.

### The four main decision outputs

```
1. MAINTENANCE SCHEDULE        2. REPAINTING TIMELINE
   Project corrosion forward      Sort zones by coating thickness
   Find date threshold crossed    URGENT = below 15 µm threshold
   Add 14-day safety buffer       WARNING = 15–25 µm, watch zone
   Output: zone + action + date   Output: priority queue + dates

3. FUEL LOSS IN DOLLARS        4. RISK ZONE MAP
   barnacle density × drag        0.4 × corrosion + 0.3 × coating
   drag × area = fuel penalty %   0.2 × fouling + 0.1 × flow stress
   × fuel cost = annual $         Score 0.0–1.0 per zone per week
   Output: live cost on dashboard Output: colour map + alerts
```

### Decision Output 1 — Maintenance Schedule

```python
for zone in ship.all_zones:
    days_to_warning = (WARNING_THRESHOLD - zone.corrosion_depth) \
                      / zone.corrosion_rate * 365
    maintenance_due_date = today + days_to_warning - 14  # 14-day safety buffer

    action = 'spot-coat'        if zone.risk_score < 0.6 else \
             'full-repaint'     if zone.risk_score < 0.8 else \
             'drydock-inspect'

# Example output:
# Zone Z_022 (Keel-Midship):
# Current depth: 1.1mm | Rate: 0.3mm/year | Failure projected: 2025-03-14
# Maintenance due: 2025-02-28 | Action: spot-coat + inspection
```

### Decision Output 2 — Repainting Timeline (Priority Queue)

```
[URGENT]  Zone Z_012 | coating:   8 µm | repaint in  3 weeks
[URGENT]  Zone Z_009 | coating:  11 µm | repaint in  5 weeks
[WARNING] Zone Z_034 | coating:  18 µm | repaint within 2 months
[WARNING] Zone Z_047 | coating:  22 µm | repaint within 3 months
[OK]      Zone Z_056 | coating:  87 µm | fine for ~14 months
[OK]      Zone Z_003 | coating: 141 µm | fine for ~25 months
```

### Decision Output 3 — Fuel Loss Due to Barnacles (in dollars)

```python
# Step 1: Drag contribution from each zone
for zone in ship.all_zones:
    zone.drag_contribution = zone.drag_coefficient * zone.area

# Step 2: Ship-level drag increase vs clean hull
total_drag            = sum(zone.drag_contribution for zone in ship.all_zones)
reference_drag        = sum(zone.base_drag * zone.area for zone in ship.all_zones)
drag_increase_fraction = (total_drag - reference_drag) / reference_drag

# Step 3: Convert to fuel penalty
fuel_penalty_percent = drag_increase_fraction * PROPULSIVE_EFFICIENCY * 100

# Step 4: Convert to annual cost
extra_fuel_tonnes = fuel_penalty_percent/100 * annual_fuel_consumption_tonnes
annual_cost       = extra_fuel_tonnes * BUNKER_FUEL_PRICE_USD_PER_TONNE

# Example output:
# Drag increase: +4.3% | Fuel penalty: +3.1% | Annual extra cost: ~$82,000
# Cost of one hull cleaning: ~$15,000 → net saving from cleaning: $67,000/year
```

### Decision Output 4 — Risk Zone Map

```python
risk_score = (  0.40 * normalise(corrosion_depth,   max=CRITICAL_CORROSION)
              + 0.30 * normalise(1 - coating_thick,  max=MAX_COATING)
              + 0.20 * normalise(barnacle_density,   max=HEAVY_FOULING)
              + 0.10 * normalise(flow_velocity,      max=MAX_FLOW) )

# Weight reasoning:
# 40% corrosion — structural damage, most dangerous and expensive
# 30% coating   — controls everything else, early warning signal
# 20% fouling   — operational cost driver (fuel)
# 10% flow stress — coating wear driver, secondary indicator
```

**Risk scale:**
`0.0–0.3 → LOW` · `0.3–0.6 → MEDIUM` · `0.6–0.8 → HIGH` · `0.8–1.0 → CRITICAL`

### Thresholds — the hard limits that trigger actions

| Variable | Warning Threshold | Critical Threshold | What Happens When Crossed |
|----------|------------------|--------------------|--------------------------|
| `corrosion_depth` | 1.5 mm | 3.0 mm | Spot repair required / structural inspection |
| `coating_thickness` | 20 µm | 10 µm | Repaint zone / emergency recoat |
| `barnacle_density` | 50 /m² | 150 /m² | Schedule hull cleaning / emergency clean |
| `fuel_penalty` | 3% | 8% | Schedule cleaning / emergency port stop |
| `risk_score` | 0.60 | 0.80 | Escalating alert / immediate intervention |

### Three alert types

**Zone Alert** — One zone's variable crosses a threshold
```
[ZONE ALERT] Zone Z_031 | corrosion_depth = 1.7mm (WARNING: 1.5mm)
Recommended action: schedule inspection
```

**Ship-Level Alert** — An aggregate ship metric crosses a threshold
```
[SHIP ALERT] Fuel penalty reached 5.2% above WARNING threshold
Cause: heavy fouling in keel zones Z_040 to Z_048
```

**Predictive Alert** — Simulation forecasts a threshold crossing within 30, 60, or 90 days
```
[PREDICTIVE] Zone Z_022 will reach CRITICAL corrosion in 47 days
Confidence: 87% | Recommended: pre-book drydock slot
```

### Optimisation outputs

```python
# 1. Optimal cleaning interval
optimal_interval_months = minimise(cleaning_cost + total_fuel_penalty_over_period)
# Example: clean every 14 months → saves net $52,000/year after cleaning cost

# 2. Route-based fouling forecast
# Route A (tropical waters): fouling rate 2.1× faster → add cleaning stop in 8 months
# Route B (North Atlantic):  fouling rate standard    → no change to schedule
```

## PHASE 06 — Real-World Integration Layer
*`integration_layer.md`*

### The problem this layer solves: drift

A digital twin that never receives real data will slowly become wrong over time. The integration layer prevents this by providing real-world corrections that keep the twin calibrated to the actual ship.

### Three input channels — how reality talks to the twin

#### Input Channel 1 — Onboard Hull Sensors

| Sensor Type | What It Measures | How Often | Feeds Into |
|-------------|-----------------|-----------|-----------|
| Corrosion probe (LPR) | Active corrosion rate at probe | Hourly | `corrosion_rate` |
| Ultrasonic thickness gauge | Actual coating + metal thickness | Daily | `coating_thickness` |
| Hull-mounted camera | Visual fouling (computer vision) | On demand | `barnacle_density` |
| Water quality sensor | Temperature, O₂, salinity at hull | Continuous | `local_*` variables |
| Speed + fuel monitor | Ship speed, fuel burn rate | Continuous | `flow_velocity`, `fuel_penalty` |

*Data path: Hull sensor → ship-side edge gateway → satellite or port WiFi → shore ingestion server → state corrector → zone state update in the twin.*

#### Input Channel 2 — Physical Inspection Reports

```python
def apply_inspection_correction(inspection_data, inspection_date):
    for zone_id, measured in inspection_data.items():
        zone = ship.get_zone(zone_id)

        # Override simulation state with measured ground truth
        zone.corrosion_depth    = measured['corrosion_mm']
        zone.coating_thickness  = measured['coating_um']
        zone.barnacle_density   = measured['fouling_count_per_m2']
        zone.last_validated     = inspection_date

        # Check if the simulation was drifting from reality
        drift = abs(zone.corrosion_depth - previous_sim_value)
        if drift > DRIFT_TOLERANCE:
            flag_model_for_recalibration(zone_id)

# Accepted formats: drydock CSV, ROV survey PDF, UT scan XML
```

*Real World → Data → Simulation → Quantum → Output*

### One-sentence summary

Real-world data (sensors, inspections, satellite) enters through the ingestion layer → gets stored in the environment database → feeds the classical simulation engine (which reads quantum-precomputed parameter tables at startup) → simulation outputs are stored in the state database → the decision engine reads these and produces maintenance schedules, repainting timelines, fuel cost estimates, and risk maps → all delivered through the dashboard, alert system, and report generator.

### Complete data handoff reference

| From | To | What Gets Sent | Format | Frequency |
|------|-----|---------------|--------|-----------|
| Hull Sensors | Ingestion Layer | Corrosion rate, coat thickness | JSON / REST | Hourly / Daily |
| Inspection Reports | Ingestion Layer | Actual measured zone values | CSV / PDF | Per inspection |
| Satellite (CMEMS) | Environment DB | Temp, salinity, O₂ grids | NetCDF→SQL | Daily |
| AIS Feed | Environment DB | Ship position, speed, port log | JSON→SQL | Hourly |
| Environment DB | Simulation Engine | Per-zone environment values | DB query | Per timestep |

| Quantum Compute | Simulation Engine | `coating_properties.json` | JSON file | On demand |
| Simulation Engine | State DB | Zone state snapshots | HDF5 | Per timestep |
| State DB | Decision Engine | Current + historical state | DB query | Per timestep |
| Decision Engine | Dashboard | Risk scores, maint. dates | REST API | Real-time |
| Decision Engine | Alert System | Threshold breach events | Email/SMS | On trigger |
| Decision Engine | Report Generator | Structured summaries | PDF / CSV | Scheduled |

### System startup sequence

```
When the system starts (first deployment or restart):

  1. Load material_table.json and coating_properties.json
     — these MUST exist before simulation can start
     — if missing, run quantum compute layer first

  2. Initialise ship zone grid
     — from last drydock inspection data (best case)
     — or from baseline estimates if first-time deployment

  3. Connect to environment database
     — verify ETL pipeline has run and data is fresh (within 24 hours)

  4. Set simulation parameters
     — start_date, end_date, timestep (default: weekly), ship_id

  5. Run simulation step cycle loop until target end date
     — in live mode: automatically advance one week every real week

  6. Start dashboard server and enable WebSocket live updates

  7. Activate alert monitoring
     — all threshold checks now live, routing matrix active
```
